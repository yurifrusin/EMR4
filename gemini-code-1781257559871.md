# EMR4 Centaur: Architecture Updates & Pivot Log

This document outlines the refined frontend architecture and workflow mechanics for the EMR4 Centaur system, replacing earlier assumptions. These pivots focus on maximizing screen real estate, streamlining the UI, and preserving flexible legacy workflows using modern MS 365 cloud capabilities.

## 1. The "Command Center" Taskpane (Dual-Monitor Architecture)
* **Concept:** Abandon the narrow 300px scrolling sidebar. The system is designed assuming the GP operates with dual monitors (or a single ultrawide).
* **Execution:** The user manually "undocks" the Word taskpane and maximizes it on their second screen. The taskpane (a borderless WebView2 browser) utilizes all available pixels.
* **UI Impact:** The frontend must be built as a fully responsive Single Page Application (SPA) using CSS Grid/Flexbox, transforming from a column into a sprawling, multi-panel dashboard when expanded.

## 2. Minimalist Word Ribbon
* **Concept:** Avoid heavily customizing the Word Ribbon XML, as native Ribbon controls are rigid and difficult to maintain.
* **Execution:** Keep the Ribbon extremely minimal (e.g., a single custom tab with a "Toggle EMR4 Copilot" button). All complex workflows, administrative tasks, and tools live natively inside the web-based taskpane.

## 3. SPA Tabbed Interface (Deprecating Floating Dialogs)
* **Concept:** Keep the user in a single, unified interface rather than relying on the Office Dialog API for popups (e.g., for Billing or Results review).
* **Execution:** The taskpane features a persistent navigation bar (e.g., `[ Consult ] [ History ] [ Billing ]`). Clicking tabs instantly swaps the DOM components via modern JS, providing a seamless, desktop-app feel without triggering browser popup blockers.

## 4. Document-Aware Routing (Contextual UI)
* **Concept:** The taskpane's available tabs and tools react to the specific Word document currently open on the primary screen via a hidden `<emr4:document-type>` Custom XML Part.
* **Execution:**
    * **Patient File Document:** Taskpane shifts to "Patient Mode" (Scribe Scribe, DDx, Meds, Demographics).
    * **Diary/Schedule Document:** Taskpane shifts to "Administrative Mode" (Waiting Room, Booking, and Daily Billing Review).
    * *Note:* Daily Billing Review is a practice-wide administrative task, so it lives exclusively in Diary Mode, separate from individual patient workflows.

## 5. Secure Role-Based Access Control (RBAC) & Dynamic UI
* **Concept:** Visual components and data access are strictly governed by the identity of the logged-in user.
* **Visual Level:** The taskpane reads a JWT upon login. If the role is "Receptionist", the JS completely omits sensitive tabs like "Billing Review" from the DOM.
* **Data Level:** The FastAPI backend enforces strict `@require_role` route guards. When a GP accesses the Billing tab, the backend queries the database using their specific `practitioner_id` from the JWT, ensuring they only review and finalize their own daily MBS claims.

## 6. The "Living Diary" (Modernizing the Macro Workflow)
* **Concept:** Recreate the flexible, free-text nature of the legacy VBA macro diary while eliminating the 30-second refresh loop and accidental deletion risks.
* **Native Co-authoring:** The daily `Diary_[Date].docx` file is hosted on Microsoft 365 (SharePoint/OneDrive). Receptionists and GPs see each other's typing in real-time, completely natively.
* **The "Parse & Lock" Workflow:**
    1.  A receptionist types a shorthand booking (e.g., `>>09:00 Jane Doe`) and hits `Enter`.
    2.  They trigger a custom Word shortcut (e.g., `Ctrl+Shift+B`).
    3.  The Add-in parses the line, pings the FastAPI backend to find the patient, registers the appointment, and converts the text into a hyperlinked block: **[09:00 - Jane Doe]**.
    4.  The Add-in wraps this paragraph in a Rich Text Content Control and applies `cannotEdit = true` and `cannotDelete = true`. The booking is now indestructible.
* **The "Unlock & Amend" Workflow:**
    1.  To fix a mistake or cancellation, the user clicks the locked block and presses a second shortcut (e.g., `Ctrl+Shift+U`).
    2.  The Add-in removes the locks, reverting it to editable text, allowing the user to amend the time or delete the row.
* **Free-Text Spaces:** Because the lock only applies to the specific paragraph containing the appointment, the blank lines between appointments remain 100% unprotected, allowing receptionists to freely type notes, messages, and reminders.
